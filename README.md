# callmap_gen.py

Статический анализатор Python-проектов, который строит карту вызовов функций — кто что вызывает и кто вызывает кого. Работает через AST, без запуска кода, не требует зависимостей кроме стандартной библиотеки.

---

## Возможности

- **Три формата вывода** — Markdown, интерактивный HTML-документ, интерактивный D3.js граф
- **Двусторонняя карта** — для каждой функции показывает исходящие вызовы (`→ вызывает`) и входящие (`← вызывается из`)
- **Умный фильтр шума** — встроенные функции (`len`, `print`, `range`…), методы стандартной библиотеки и вызовы на локальных переменных (`ticker.get()`, `row.items()`) отфильтровываются автоматически
- **Разрешение импортов** — алиасы (`import storage as db`) раскрываются до исходного модуля, ссылки между файлами кликабельны
- **Поддержка `async`**, методов классов, вложенных функций, декораторов
- **Без зависимостей** — только стандартная библиотека Python 3.10+

## Установка

Скопируйте `callmap_gen.py` в любое удобное место. Дополнительных пакетов не требуется.

```bash
python callmap_gen.py --help
```

---

## Использование

```bash
# Markdown (по умолчанию)
python callmap_gen.py ./myproject

# HTML-документ
python callmap_gen.py ./myproject --format html

# Интерактивный граф
python callmap_gen.py ./myproject --format graph

# Все три формата сразу
python callmap_gen.py ./myproject --format all -o docs/callmap

# Формат по расширению файла
python callmap_gen.py ./myproject -o docs/callmap.html
python callmap_gen.py ./myproject -o docs/callmap.graph.html
```

## Параметры CLI

| Параметр | Описание |
|---|---|
| `project` | Путь к корню проекта (обязательный) |
| `-o, --output` | Файл вывода. Формат определяется по расширению `.md` / `.html`. По умолчанию `call_map.md` |
| `--format` | Явно задать формат: `md`, `html`, `graph`, `both` (md+html), `all` (все три) |
| `--exclude` | Папки для исключения через запятую, добавляются к стандартным |
| `--min-calls N` | Показывать только функции с ≥ N исходящими вызовами |
| `--no-locals` | Не включать вложенные функции (closure) |

По умолчанию исключаются: `__pycache__`, `.git`, `venv`, `.venv`, `node_modules`, `dist`, `build`, `migrations`, `.mypy_cache`, `.pytest_cache`.

## Форматы вывода

### Markdown (`.md`)

Статический документ в стиле GitHub. Содержит:
- Оглавление с кликабельными ссылками на каждый файл
- Для каждой функции — список исходящих вызовов, сгруппированных по модулю, и список функций, которые её вызывают
- Ссылки между файлами через якоря (`#file-...`)

Удобен для хранения в репозитории рядом с кодом.

### HTML-документ (`.html`)

Интерактивный документ с тёмной темой. Содержит:
- Боковую панель с навигацией по файлам и подсветкой текущего раздела при прокрутке
- Поиск в реальном времени — по файлам в сайдбаре и по процедурам в основной части
- Сворачивание отдельных функций кликом по заголовку
- Плашки «Вызывается из» с кликабельными ссылками прямо на карточку вызывающей функции

### Граф (`.graph.html`)

Интерактивный D3.js граф с двумя режимами, переключаемыми кнопками в топбаре.

**Режим «Функция → Функция»** (детальный):

- Узлы сгруппированы по файлу — каждый файл обведён полупрозрачной оболочкой с подписью (кнопка ▦ — включить/выключить)
- Цвет узла кодирует тип: 🟢 зелёный — root-функция (никто не вызывает), 🔵 синий — обычная, 🟣 фиолетовый пунктир — `async`
- Размер узла пропорционален суммарному числу входящих и исходящих вызовов
- Цветные стрелки при hover/выборе: зелёные — исходящие, синие — входящие
- **Клик по узлу** — открывает боковую панель со списком «Вызывает» и «Вызывается из»; клик по имени в панели — переход к той функции с автопаном
- **Изолировать** — кнопка в панели гасит всё кроме выбранной функции и её непосредственных соседей
- **Фильтр по файлу** — дропдаун перерисовывает граф только для функций одного файла

**Режим «Файл → Файл»** (обзорный):

- Узлы — файлы проекта, рёбра — вызовы между файлами
- Толщина ребра пропорциональна количеству вызовов между файлами

**Общие для обоих режимов:**

- Зум колёсиком и кнопками ＋ / － / ⌂
- Перетаскивание и фиксация узлов (клик на зафиксированный снимает фиксацию)
- Поиск по имени — подсвечивает совпадающие узлы

## Как это работает

1. **Сканирование** — рекурсивный обход `*.py` файлов проекта через `pathlib`
2. **Парсинг** — каждый файл разбирается через `ast.parse`; собираются импорты, функции, методы классов и вложенные closure
3. **Разрешение вызовов** — для каждого `ast.Call` определяется целевой модуль через таблицу импортов. Вызовы на неимпортированных объектах (локальных переменных) помечаются маркером `_VAR_` и отфильтровываются
4. **Построение обратного индекса** — после сбора всех вызовов строится индекс `callee → [callers]`
5. **Генерация** — данные передаются в рендерер выбранного формата

## Ограничения

- Анализ статический — динамические вызовы (`getattr(obj, name)()`, `func_map[key]()`) не отслеживаются
- Только Python — другие языки не поддерживаются
- Вызовы через декораторы-обёртки могут не разрешиться до оригинальной функции
